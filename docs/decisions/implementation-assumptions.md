# Web Demo 实施假设

## 已采用的 Demo 级假设

1. 项目开始时没有完整前后端技术栈。当前长期方向调整为 Next.js App Router + Supabase 的全栈一体架构；本轮 Demo 继续使用 React + TypeScript + Vite 与本地状态层，未来迁移到 Next.js 时不改变产品状态和 Mock 契约。
2. 登录注册使用本地模拟邮箱表单，不建立真实身份，也不发送短信。
3. 用户甲为丹青、用户乙为林夏；两人是熟悉的家人，示例事件为一次温和的冬至晚饭。
4. Demo 语音、转写、OCR、图片理解、插图、TTS 和交汇均使用前端模拟；原始内容和 AI 结果的可见性仍按正式权限规则执行。
5. 预约流程只实现已约定时间的设置、修改、取消和“到时”模拟；复杂双方协商及跨时区暂不实现。
6. 专属表情在 Demo 中同时作为主要回应和阅读确认；是否另设独立已读按钮仍待产品确认。
7. 送达前取回次数、共同内容删除和账号注销规则未决，因此不提供可能误导的正式删除能力。
8. Demo 交汇可以从控制台快速推进，但产品页面仍执行“双方正式信件＋双向已读”的资格判断。
9. 记忆树的浇水状态保存在本地，不包含提醒、积分、连续天数或枯萎。

## 文档表述差异

- PRD 对共同事件卡最低字段定义较轻，完整 Demo 任务额外要求“相关人物”和“想一起回看的原因”；实现增加这两项，不改变中性化和隐私规则。
- PRD 的页面地图没有单独定义产品首页，但完整 Demo 任务明确要求交换列表首页；实现增加 `/home` 作为已登录后的主要入口。
- 长期架构文档建议由 Next.js 服务端承载业务命令、权限校验和 AI 网关；当前 Demo 不调用真实 AI。所有 Mock 只在浏览器中运行，真实替换点仍保持服务端接口边界。

这些假设不删除或弱化 PRD 的核心流程。
