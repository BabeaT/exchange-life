# Web Demo 范围与开发规划

## 1. Demo 最短完整流程

以“共同事件＋双方独立表达”为主演示路径，同时用组合选择和测试数据验证另外三种组合可成立：

1. A 从品牌入口注册/登录并创建资料。
2. A 选择共同事件、填写并确认事件卡，选择双方独立表达和立即送达。
3. 系统生成邀请码；B 登录后输入邀请码并加入双人空间。
4. A、B 分别使用文字完成私人表达。
5. 两人分别查看模拟 AI 整理稿、编辑并本人确认。
6. 两人预览并寄出；先寄出的信被托管，双方均寄出后同时送达。
7. 双方主动开启并阅读对方信件，分别发送一个专属表情确认已读。
8. 系统基于两封最终公开信件生成模拟 AI 交汇。
9. 双方查看共同记忆碎片。

另需一条短分支验证“我先告诉对方”：A 寄出后 B 可只发表情结束，不被强制回信，也不生成交汇碎片。

## 2. 可使用模拟数据的功能

- AI 事件卡提取、叙事整理、插图、交汇结果。
- 语音转写、方言识别、OCR、图片理解和 TTS。
- 记忆信使动画、专属表情动作和记忆树视觉生长。
- 预约时间推进、网络失败、AI 失败等异常场景。
- Demo 演示账号与预置历史记忆。

模拟结果仍必须经过本人确认界面，交汇模拟数据也只能引用演示中的最终公开信件。

## 3. 必须接入真实数据的功能

“真实”指 Demo 运行时必须有可验证的持久状态，不等同于本阶段立即连接 Supabase：

- A/B 两个独立身份或可明确切换的隔离会话。
- 唯一交换、邀请码领取和双人绑定关系。
- 两个独立选择维度及四种组合值。
- 私人草稿隔离、本人确认状态和最终公开信件版本。
- 托管/送达状态、阅读确认、主要表情回应。
- 交汇资格判断和共同记忆碎片的创建条件。

在接入 Supabase 前可用本地 Demo 数据层实现，但不能只靠静态页面假装状态转换。

## 4. 第一阶段暂缓

- 真实 Supabase、外部 AI、语音识别、OCR、图片生成和 TTS 服务接入。
- 完整录音编辑、方言校正、图片素材管理和公开附件流程。
- 预约交换协商、送达前取回次数配置及完整冲突处理。
- 信使性格测试、多套动画资产和完整表情资产。
- 记忆树浇水、同时在线动画、季节氛围、微光与原句回响。
- 复杂删除、数据导出、账号恢复和生产级审计后台。

## 5. 建议开发顺序与验收条件

| 阶段 | 范围 | 验收条件 |
|---|---|---|
| 0. 文档与模型冻结 | 核心规则、页面图、状态机、数据草案 | 四种组合无冲突；隐私、AI 确认和交汇来源规则可追踪 |
| 1. 应用骨架与身份模拟 | 路由、桌面布局、A/B 隔离会话 | 可从入口进入创建或邀请码目标；刷新后业务目标不丢失 |
| 2. 创建与邀请 | 两维度选择、事件/主题卡、确认、邀请码 | 四种组合均可创建；邀请码仅绑定一个非本人 B；B 只读已确认前置卡 |
| 3. 私人表达与确认 | 文字草稿、保存、模拟 AI 整理、差异与确认 | A/B 草稿互不可见；未确认内容不能预览或寄出；修改使旧结果失效 |
| 4. 信件与送达 | 预览、寄出、托管、单方/双向送达 | 独立表达先寄信不可提前阅读；单方分享不等待回信；状态转换可恢复且幂等 |
| 5. 阅读与回应 | 主动开启、阅读位置、专属表情、可选回信 | 表情确认已读且每信一个主要回应；B 可不回信，无催促状态 |
| 6. 交汇与记忆 | 资格校验、模拟交汇、碎片 | 只有双方最终公开信及双向已读才生成；来源不包含任何私人材料 |
| 7. 异常与可访问性 | 失败恢复、加载状态、键盘与减少动态 | 关键失败不丢原始内容；动画可跳过；音频不自动播放；桌面常见视口可用 |
| 8. Supabase/AI 接入（后续） | 按批准范围替换模拟适配器 | RLS 与服务端校验通过；密钥不在浏览器；真实 AI 仍需本人确认 |

## 6. Demo 总体验收

- 最短流程可由两名隔离用户从创建走到共同记忆碎片。
- 四种组合的选择、文案、送达和交汇条件均与需求一致。
- 原始内容、草稿、AI 摘要默认仅本人可见。
- 所有 AI 候选结果都有本人确认门槛。
- 单方分享允许只发表情结束；没有聊天、匹配、排行、签到或树木枯萎机制。

## 7. 仍待产品决策

以下问题保留为决策项，本文件不替产品负责人定案：

1. 注册登录使用邮箱、手机号、第三方登录中的哪一种或哪些组合？
2. 邀请码默认有效期、失效后是否允许 A 重新生成，以及交换取消后的展示方式是什么？
3. B 无法识别共同事件时，A 可补充的线索格式、次数和隐私审核规则是什么？
4. 预约送达的发起、确认、修改、取消和时区规则如何定义？
5. 送达前取回是否支持、允许次数，以及对方能否看到“曾取回”的粗粒度提示？
6. “完成阅读确认”除发送专属表情外，是否还需要无情绪含义的独立已读按钮？
7. 回信是否必须沿用原叙事内容和前置卡，还是允许在关联下选择新的主题？
8. AI 交汇由系统自动触发还是由任一方主动触发？重生成次数与双方结果一致性如何处理？
9. 双方对交汇问题段落意见不一致时，隐藏范围和共同版本如何处理？
10. 已送达信件、公开附件、交汇与共同记忆碎片的删除、撤回、隐藏和账号注销规则是什么？
11. 原始录音、图片、AI 中间结果和生成音频的保存期限及用户导出能力如何定义？
12. Demo 是否必须真实接入身份与持久化，还是允许用隔离的本地数据层完成首次评审？
