# 《交换人生》功能需求说明

**核心范围：**双人叙事 · 延迟交换 · 视角交汇 · 共同记忆

> 本文件是VS Code与Codex开发时使用的功能需求依据。Word版用于交付与阅读，本Markdown版用于代码实现、检索和持续更新。

# 文档范围

本文件只描述《交换人生》Web端的功能需求、业务规则、页面状态、内容权限和验收要求，不包含产品背景、市场分析、竞品分析、用户研究、商业模式、技术架构和视觉风格定稿。

> **核心边界**
> “共同事件/一段人生”决定讲述范围；“双方独立表达/我先告诉对方”决定参与方式。两个维度相互独立，可自由组合。

> **2026-07 产品澄清**
> 本文原有“人生片段”统一理解为“一段人生/人生周期”。它不是一篇一次性长文，而是有开始与结束条件、允许多次记录的私人时间线。单方模式根据一人的确认内容生成个人故事与绘本；双方模式先分别确认并提交故事源稿，等两边都提交后再生成共同故事与共同绘本，不提前生成或送达任何一方的个人绘本。完整产品模型、交互调整和未决问题见 `docs/product/complete-prd.md`。

> **产品定位**
> 信件是视角传递的载体，不是产品目的。核心行为是双方独立表达、个人视角抵达、阅读时暂时转换观看位置、AI整理公开视角的交汇，以及在保留双方独立身份和感受的前提下形成共同记忆。产品不承诺和解，不判断哪一方正确，也不消除既有差异。

# 1. 全局业务规则与状态定义

## 1.1 核心参与者

| **角色** | **定义**                                     | **主要权限**                                                            |
|----------|----------------------------------------------|-------------------------------------------------------------------------|
| 发起人A  | 创建一次交换并生成邀请码的用户               | 选择叙事内容与交换方式；创建事件卡或主题卡；邀请B；完成自己的表达与寄出 |
| 受邀人B  | 通过熟人私下发送的邀请码进入产品的用户       | 加入指定双人交换空间；查看前置卡；完成表达或阅读来信；自主决定是否回信  |
| AI       | 提供整理、识别、图像生成、读信与视角交汇能力 | 只能在授权范围内处理内容；不得代替本人确认，不得跨越私人内容权限        |

## 1.2 四种组合模式

| **讲述范围** | **参与方式** | **核心流程** | **最终结果** |
|--------------|--------------|--------------|--------------|
| 共同事件 | 双方独立表达 | 围绕同一事件盲写，双方确认源稿后同时揭晓并共同生成 | 两份源稿、共同故事、双视角绘本 |
| 共同事件 | 我先告诉对方 | A 讲述并确认，生成个人绘本后送达 B | A 的个人故事与绘本；B 可选回写 |
| 一段人生 | 双方独立表达 | 在约定周期内分别记录，周期结束且双方提交后共同生成 | 双轨时间线、共同故事、共同绘本 |
| 一段人生 | 我先告诉对方 | A 在一个周期内持续记录，完成后生成并送达 | A 的人生时间线与个人绘本 |

## 1.3 关键状态定义

| **状态**          | **定义**                                       | **对方是否可见内容**                             |
|-------------------|------------------------------------------------|--------------------------------------------------|
| 编辑中            | 用户正在以文字、语音、图片表达，系统自动保存   | 不可见                                           |
| 待整理            | 用户结束本轮表达，等待AI生成整理稿             | 不可见                                           |
| 待本人确认        | AI整理稿、个人绘本或共同绘本等待核对和确认     | 不可见                                           |
| 已寄出            | 用户确认内容并提交，信件进入系统托管或配送流程 | 双向独立模式下不可见；单方分享模式按送达规则处理 |
| 等待对方          | 一方已寄出，另一方尚未寄出                     | 不可见具体进度                                   |
| 待送达            | 已满足双方寄出或预约时间条件                   | 不可见                                           |
| 已送达            | 信使送信动画可触发，收信人尚未打开             | 可见信封，不可见正文                             |
| 阅读中/已读完     | 收信人已打开或完成阅读确认                     | 寄信人仅获知已读状态和专属表情                   |
| 交汇生成中/已完成 | AI依据双方公开信件生成交汇结果                 | 完成后双方共同可见                               |

## 1.4 全局原则

- 独立表达期间，双方互不可见对方草稿、录音、图片、AI摘要和完成进度细节。

- AI整理后的内容必须经本人确认；未确认内容不得寄出、不得被对方查看、不得进入视角交汇。

- 双方独立表达时，一方可先寄出，但系统须托管信件，直至双方均寄出或约定送达时间满足。

- “我先告诉对方”不要求收信人回信；专属IP表情即可构成最低限度的已读回应。

- 只有存在双方最终确认并公开的信件时，才生成AI视角交汇和共同记忆碎片。

- AI交汇只基于最终公开信件，不得读取、引用或泄露任何一方未公开的原始材料。

- 双方独立表达中，双方各自只确认故事源稿；共同故事与共同绘本必须等双方提交后生成，并由双方校对确认后归档。

- “一段人生”采用可多次新增记录的私人时间线，单条记录保存或完成不等于提交整个项目。

- 所有关系空间以双方名字呈现，不强制使用“母女、情侣、朋友”等关系标签。

# 2. 品牌入口页

### FR-ENT-001 · 页面定位

品牌入口页仅承担品牌出现、氛围建立和进入产品的作用，不展示登录表单、邀请码、功能介绍、营销信息或复杂导航。

### FR-ENT-002 · 页面元素

首屏仅包含《交换人生》品牌名称或Logo、品牌场景图像、一个可点击的进入元素及极轻量环境动效。

品牌标题“交换人生”四字必须保持同一行，不插入空格或换行标签；使用响应式字号、足够容器宽度、`white-space: nowrap` 与 `word-break: keep-all` 等方式保证常见 Web 宽度不换行，并在窄屏缩小字号且不得造成横向滚动。

首屏主文案固定为：

> 我的话语，一半毫无意义；  
> 但我说出来，是为了让另一半能够抵达你。

页面不展示旧版底部说明文案，删除后不得保留空占位或额外底部间距。

### FR-ENT-003 · 进入交互

用户点击场景中的门、信件、光点或“进入”后进入“进入方式选择页”。页面不得要求滚动。

### FR-ENT-004 · Web端适配

参考移动端开屏氛围重新构成横向场景，不得直接等比放大手机界面；常见桌面视口中核心元素须完整可见。

### FR-ENT-005 · 动效边界

动效不得阻塞点击，不自动播放声音，支持减少动态效果；加载失败时仍保留品牌名称和进入按钮。

# 3. 进入方式选择、登录注册与个人资料

## 3.1 进入方式选择页

### FR-AUTH-001 · 双入口

页面提供“创建一次交换”和“输入邀请码”两个同级业务入口；可在角落提供低层级“已有账号？登录”。

### FR-AUTH-002 · 状态判断

任何业务入口被点击后，系统先判断当前账号状态：已登录、已注册但未登录、未注册。

| **用户状态** | **创建一次交换**                     | **输入有效邀请码**                       |
|--------------|--------------------------------------|------------------------------------------|
| 已登录       | 进入创建交换流程                     | 直接加入并进入对应交换空间               |
| 已注册未登录 | 登录后进入创建交换流程               | 暂存邀请码，登录后进入对应空间           |
| 未注册       | 注册并创建个人资料后进入创建交换流程 | 暂存邀请码，注册并创建资料后进入对应空间 |

## 3.2 注册与登录

### FR-AUTH-003 · 注册方式

注册页应提供项目确定的账号注册方式；注册成功后必须建立可恢复的登录会话。具体账号载体可在技术方案中确定，但不得影响邀请码回跳。

### FR-AUTH-004 · 业务回跳

登录或注册完成后，系统必须回到用户进入身份流程前的业务目标，不得统一跳转普通首页。

### FR-AUTH-005 · 邀请码暂存

邀请码验证成功后进入登录注册流程时，须持续保存邀请码与交换ID；刷新或完成资料后无需再次输入。

## 3.3 创建个人资料

### FR-PRO-001 · 首次创建

仅首次注册且资料未完成时展示个人资料页；老用户不重复填写。

### FR-PRO-002 · 头像

支持系统预设头像、上传本地图片和默认头像；用户可稍后修改。

### FR-PRO-003 · 昵称

昵称必填，用于邀请、信件署名、双人空间与记忆树；提示“这是对方在交换中看见的名字”。

### FR-PRO-004 · 完成条件

昵称通过长度与敏感字符校验后方可完成；完成后自动进入原业务路径。

# 4. 创建交换：双维度选择

## 4.1 第一步：选择叙事内容

### FR-CRT-001 · 共同事件

选项文案表达“重新讲述你们共同经历的一件事”；选择后进入共同事件卡。

### FR-CRT-002 · 一段人生

选项文案表达“记录并交换一段时间里的生活”；选择后进入一段人生周期卡。

## 4.2 第二步：选择交换方式

### FR-CRT-003 · 双方独立表达

选项文案为“我们都写下自己的版本”；明确说明交换开启前彼此不可见。

### FR-CRT-004 · 我先告诉对方

选项文案为“我想先把这一段告诉TA”；明确说明对方可以收下、用表情回应，也可以在准备好后回信。

### FR-CRT-005 · 组合自由

共同事件与一段人生均可选择任一参与方式，不得将一段人生固定为单向分享，也不得将共同事件固定为双向盲写。

## 4.3 创建确认

### FR-CRT-006 · 结果预览

生成邀请前展示用户已选择的内容类型、交换方式、送达方式和前置卡摘要，并允许返回修改。

### FR-CRT-007 · 创建成功

创建成功后生成唯一交换ID、邀请码和双人空间；A可复制邀请，同时可立即开始自己的表达。

# 5. 共同事件卡

### FR-EVT-001 · 创建权限

共同事件卡只由发起人A创建和确认；B只能查看，不可直接修改。

### FR-EVT-002 · 核心问题

以“你想和对方重新讲述哪一件事？”作为主要输入，优先支持自然语言描述，而非复杂表单。

### FR-EVT-003 · 卡片字段

至少包含事件名称与事件描述；大致时间、地点、事件起止范围和补充线索为选填。

### FR-EVT-004 · 时间表达

支持精确日期、大致时期、人生阶段、相对时间与“记不清具体时间”；自然语言为主，日期控件为辅。

### FR-EVT-005 · AI提取

AI从描述中提取事件主题、时间、地点、人物和事件边界；缺少必要信息时最多提出一至两个轻量问题。

### FR-EVT-006 · 中性化处理

AI须区分客观事件线索与A的主观判断；共享卡不得把“你不关心我”等单方感受整理成共同事实。

### FR-EVT-007 · 本人确认

A可修改AI整理后的事件卡，确认后生成邀请；未确认卡片不得对B展示。

### FR-EVT-008 · B端识别

B加入后先查看事件卡，并点击“我知道是哪件事，开始讲述”；B只看中性事件卡，不看A的个人草稿。

### FR-EVT-009 · 无法识别

B可选择“我还不确定是哪件事”；系统允许稍后继续，并可预留请求A补充一条事件线索的能力。

# 6. 一段人生周期卡

### FR-THM-001 · 卡片定位

主题卡用于为两段不同人生提供共同方向，不要求双方讲同一件事，不要求提前填写时间、地点或人物。

### FR-THM-002 · 预设主题

提供兼具温暖、成长、选择与失落等不同情绪方向的主题，如离开、回家、成为自己、没有说出口的话、被爱着的时刻。

### FR-THM-003 · 自定义主题

用户可输入自定义人生主题；AI可将过宽主题轻量收束，但不得改变原意。

### FR-THM-004 · 创建权限

主题卡由A创建并确认，B加入后可查看；双方独立表达模式下，B只能看到主题卡，不能看到A的故事。

### FR-THM-005 · 表达引导

进入表达页时提示“这一次，你想把人生中的哪一部分交给对方？”；时间和事件线索在自然叙述中由AI提取。

### FR-THM-006 · 周期边界

人生周期卡须包含开始时间与结束条件。结束条件可以是具体日期或由发起人手动结束；双方独立表达模式下，B 加入前须看到该条件。

### FR-THM-007 · 私人时间线

双方分别拥有一条私人时间线，可在周期内多次新增文字、语音和图片记录。每条记录支持大致时间、可选地点、标题、说明和是否纳入最终故事。

### FR-THM-008 · 记录与提交分离

保存或完成一条记录不代表提交整个项目。只有用户在周期汇总页明确确认故事源稿后，内容才进入托管或个人绘本生成。

### FR-THM-009 · 双轨隐私

双方独立表达期间，任何一方都不能查看对方的记录内容、数量、具体日期或最近编辑时间；只展示“记录中/已提交”等粗粒度状态。

### FR-THM-010 · 周期结束汇总

周期结束后，系统按时间、阶段或主题汇总本人记录。用户可排除记录、调整顺序、补充结束语，并确认最终故事源稿。

### FR-THM-011 · 到期未完成

周期到期但任一方未提交时，不自动公开已提交内容。系统提供延期、继续等待、各自保存或由已提交者重新确认为单方分享；任何模式变化不得公开另一方材料。

# 7. 邀请码与双人空间连接

### FR-INV-001 · 使用场景

邀请码由熟人通过微信、QQ等外部社交平台私下发送，不承担陌生人匹配和信任建立功能。

### FR-INV-002 · 邀请内容

系统生成可复制的邀请文案、邀请码和Web链接；Demo阶段至少实现一键复制，不强制接入外部分享SDK。

### FR-INV-003 · 有效码处理

用户输入有效邀请码并完成必要登录注册后进入邀请预览。单方模式可直接加入并等待内容；双方独立表达模式须由 B 明确点击“接受并写下我的版本”后绑定为共同创作者。

### FR-INV-004 · 异常校验

处理邀请码不存在、格式错误、失效、已被其他账号使用、输入自己邀请码、交换已取消或结束等状态。

### FR-INV-005 · 唯一绑定

一次双人交换只允许A与一个B绑定；绑定成功后，邀请码不可再次被其他账号使用。

### FR-INV-006 · 隐私展示

邀请文案仅展示完成进入所需的信息，不包含A的草稿、AI摘要、敏感图片或未公开故事内容。

# 8. 独立表达工作区

### FR-WRT-001 · 私人空间

双方分别拥有独立表达区；交换开启前互不可见内容、输入方式、草稿版本和具体进度。

### FR-WRT-002 · 多次进入

支持用户分多次完成表达；离开、刷新或重新登录后恢复到上次编辑位置。

### FR-WRT-003 · 自动保存

文字、语音片段、转写文本、图片、图片说明和编辑位置自动保存；展示轻量保存状态。

### FR-WRT-004 · 混合表达

文字、语音和图片可以任意组合并共同进入同一故事上下文，不要求用户预先选择唯一输入方式。

### FR-WRT-005 · 结束本轮

提供“先停在这里”和“完成表达”两种动作；前者保存后可继续，后者进入AI整理。

### FR-WRT-006 · 重新打开

已进入整理阶段但尚未寄出的用户，可返回补充内容；补充后原整理稿标记为需更新。

### FR-WRT-007 · 进度隐私

对方只能看到“正在准备/已寄出/已送达”等粗粒度状态，不显示字数、录音时长、最近编辑时间。

# 9. AI陪伴IP与上下文引导

## 9.1 三种状态

| **状态** | **表现**                                       | **触发**                        |
|----------|------------------------------------------------|---------------------------------|
| 安静陪伴 | IP位于编辑区边缘，仅有呼吸、眨眼或微光，不说话 | 用户正常表达                    |
| 轻提示   | 出现小气泡，如“需要我帮你理一理吗？”           | 已有有效内容且连续停顿10—15分钟 |
| 展开引导 | 展开小型侧栏，提供总结、梳理、一个小提示或暂停 | 用户主动点击IP或提示气泡        |

### FR-AIC-001 · 默认安静

AI不得在用户持续表达时自动追问、评价、续写或分析情绪。

### FR-AIC-002 · 主动唤醒

用户可随时点击IP，选择“回顾刚才写了什么、理一理回忆、给一个小提示、先停一会儿”。

### FR-AIC-003 · 停顿条件

自动提示仅在已有有效内容、页面处于前台、未录音、当前停顿未提示过且用户未关闭提醒时触发。

### FR-AIC-004 · 提示消失

用户继续输入、关闭或忽略后气泡淡出；同一停顿不重复提示，支持“本次不再提醒”。

### FR-AIC-005 · 上下文总结

AI可整理用户已经表达的主题、节点和停留位置；摘要仅本人可见，不自动进入最终信件。

### FR-AIC-006 · 轻量问题

用户主动请求后，一次只提供一个问题；问题用于恢复叙述，不要求回答，不暗示对方动机。

### FR-AIC-007 · 情绪边界

AI可使用“可以慢一点、内容已经保存”等陪伴表达，不得声称完全理解、心理诊断、评判关系或催促完成。

# 10. 多模态输入与原始材料保留

## 10.1 文字输入

### FR-MOD-001 · 文字编辑

支持连续输入、分段、多次续写、自动保存、撤销、删除和与语音转写混合编辑。

### FR-MOD-002 · 版本保留

至少保留用户原始输入、当前编辑稿和AI整理稿三个层级，AI不得覆盖原始版本。

## 10.2 语音讲述

### FR-VOI-001 · 首次引导

首次启用录音时，由IP提供“可以像跟我聊天一样说，不用着急，说错也没关系”的短示例；可跳过或不再提示。

### FR-VOI-002 · 录音准备

用户点击语音入口后先进入准备状态，取得麦克风授权并由用户再次点击开始；示例播放期间不得录音。

### FR-VOI-003 · 停顿陪伴

录音中出现较长静音时继续录音，不自动结束、不播放声音，只显示“不用着急，我在听”等小字。

### FR-VOI-004 · 长静音

静音明显过长时询问“继续录音/先停在这里”；短暂停顿不得被判定为情绪问题。

### FR-VOI-005 · 方言识别

支持自动识别与手动选择普通话、粤语和其他方言；不确定内容须标记并允许用户听原音校正。

### FR-VOI-006 · 双层文本

方言处理可生成贴近原话的转写和便于阅读的书面版本；书面版本不得覆盖原始转写。

### FR-VOI-007 · 回放默认关闭

录音完成后不自动播放，优先展示转写文本；用户主动点击才播放自己的原音。

### FR-VOI-008 · 原音保存

保存原始录音、时间、转写和修订结果；用户可播放、暂停、定位和删除单段录音。

## 10.3 图片输入

### FR-IMG-001 · 素材类型

支持旧照片、手写信、日记、聊天截图、票据、通知书、场景照片等记忆材料。

### FR-IMG-002 · 文字识别

对含文字图片执行OCR或手写识别，生成可编辑文本并标记低置信内容；始终保留原图。

### FR-IMG-003 · 视觉理解

AI可识别场景、物件和显式时间线索，但不得仅凭画面推断人物关系、心理或事件真相。

### FR-IMG-004 · 用户解释

上传后允许用户用文字或语音说明“这张图片对故事意味着什么”，解释与图片绑定保存。

### FR-IMG-005 · 公开选择

原图默认仅本人可见；用户可选择插入最终信件、仅作私人参考或用AI插图替代。

# 11. AI叙事整理与本人确认

### FR-ORG-001 · 处理时机

每一方完成表达后即可独立进入AI整理，不需要等待对方完成。

### FR-ORG-002 · 输入范围

整理依据包括共同前置卡、原始文字、语音转写、图片识别文本、图片说明和用户确认保留的内容。

### FR-ORG-003 · 允许调整

AI可去除无意义口头重复、合并重复内容、调整叙事顺序、补标点、分段并把口语转成可读文字。

### FR-ORG-004 · 禁止行为

AI不得编造情节、对白、日期、地点或动机，不得擅自美化、弱化或扩大情绪。

### FR-ORG-005 · 保留不确定性

原表达中“记不清、可能、大概”等不确定信息必须保留，AI不得改写为确定事实。

### FR-ORG-006 · 确认工作区

同时提供原始表达、AI整理稿和差异查看；整理稿可直接编辑、恢复原话、删除、补充或局部重整。

### FR-ORG-007 · 确认动作

用户点击“这就是我想说的”后，文字才成为故事源稿。单方模式可继续生成个人绘本；双方模式进入托管并等待另一方源稿。未确认稿不得提交或寄出。

### FR-ORG-008 · 修改联动

确认后若重新修改正文，相关摘要、插图和版式标记为需同步更新。

# 12. AI故事与绘本生成

### FR-LET-001 · 生成顺序

先确认文字，再生成插图和版式，避免文字反复修改导致图文不一致。

### FR-LET-002 · 关键画面

AI从确认文字中提出房子、旧信封、车站、行李、灯光等关键场景或物件，用户确认后生成。

### FR-LET-003 · 图像真实性

AI插图须标识为叙事插图；优先非写实、绘本或象征风格，不把生成细节宣称为真实记忆。

### FR-LET-004 · 人物边界

未提供参考时不擅自生成可识别真人面孔；涉及具体人物、地点和敏感情节时须由用户确认。

### FR-LET-005 · 图文比例

提供“文字为主、图文平衡、图像为主”三个预设；用户可调整图片数量、位置、删除和重新生成。

### FR-LET-006 · 文字完整性

改变图文比例不得静默删除已确认内容；图像为主需要精简文字时，须生成新版本并再次由本人确认。

### FR-LET-007 · 原图使用

用户可用本人上传图片替换AI插图，并明确选择该原图是否向对方公开。

### FR-LET-008 · 完整预览

寄出前提供收信人视角的完整预览，包括封面、正文、插图、署名和可公开附件。

### FR-LET-009 · 单方生成

“我先告诉对方”模式在作者确认故事源稿后生成个人故事与绘本。AI 只可使用作者明确纳入最终故事的内容；作者确认完整预览后方可寄出。

### FR-LET-010 · 双方延后生成

“双方独立表达”模式不在单方提交后生成个人绘本。双方分别确认故事源稿并托管，待两边都提交后锁定公开快照，再生成共同故事与共同绘本。

### FR-LET-011 · 共同结果结构

共同结果须分别保留双方视角、共同线索和仍有差异或无法确认的部分。每个事实性段落须能追溯到至少一份公开源稿，不得编造统一结论。

### FR-LET-012 · 双方校对

双方均可对共同故事和插图标记“事实不准确、混淆双方、语气不属于我或不适合共同公开”。共同绘本只有在双方确认后才进入共同记忆；未双确认版本保持待校对状态。

### FR-LET-013 · 源稿访问策略

双方模式保存 `source_access_policy`。`hidden` 表示双方始终不能在产品中阅读彼此源稿；`after_storybook` 表示当前用户完成共同绘本阅读后才显示源稿入口。无论使用哪种策略，系统都须保留仅服务端和共同 Agent 可用的来源引用，用于事实校验和问题反馈。

# 13. 记忆信使与形象测试

### FR-MSG-001 · 功能定位

记忆信使是用户在产品内的个人IP形象，用于陪伴书写、送信和表情回应；与头像并存。

### FR-MSG-002 · 获取方式

用户可完成简短测试、直接选择预设形象、使用默认信使或稍后再设置；不得阻塞注册和寄信。

### FR-MSG-003 · 测试边界

测试定位为记忆与表达偏好匹配，不宣称心理诊断，不区分优劣，不影响AI对故事的判断。

### FR-MSG-004 · 建议维度

可围绕回忆入口、表达方式、关系节奏、整理方式四个双向维度组合固定结果；具体题目与形象数量在专项设计中确定。

### FR-MSG-005 · 固定形象

每种形象具有名称、外观、颜色、陪伴动作、送信动作和简短说明；用户可查看并更换。

### FR-MSG-006 · 资产复用

不同信使优先复用统一动作骨架，仅替换外观与装饰，降低多形象×多表情的动画制作量。

# 14. 寄出、托管、预约与送达

## 14.1 确认寄出

### FR-SND-001 · 最终确认

用户完成图文预览后点击“确认寄出”；该动作同时代表内容确认和交换意愿，无需再次点击“同意交换”。

### FR-SND-002 · 寄出提示

明确说明寄出后何时送达、对方是否立即可见，以及是否可以在送达前取回修改。

### FR-SND-003 · 送达前取回

建议允许用户在正式送达前取回修改；取回后恢复编辑中并重新满足寄出条件。具体次数可配置。

## 14.2 双方独立表达

### FR-SND-004 · 系统托管

任一方可先寄出，系统托管信件；对方在自己寄出前不得阅读先寄出的信。

### FR-SND-005 · 立即交换

默认在双方均提交源稿后锁定两份快照并直接生成共同故事与共同绘本，两人无需同时在线；生成前不向双方揭晓彼此源稿。

### FR-SND-006 · 预约交换

双方可约定具体送达时间；一方提出、另一方确认，双方可提前寄出。

### FR-SND-007 · 预约未满足

约定时间到达但一方未寄出时，不得单独送达另一封信；提示继续等待或重新约定，不透露具体写作进度。

## 14.3 我先告诉对方

### FR-SND-008 · 单方送达

A寄出后可立即或预约送达B，不以B回信为前置条件。

### FR-SND-009 · 无回信压力

系统不向B显示倒计时、等待提示或未完成任务；A只看到信件已送达、已打开或已被表情回应。

### FR-SND-010 · 稍后回信

B可在任意时间从收到的信件进入“写一封回信”；原信与主题上下文自动关联。

# 15. 信件送达、开启与语音读信

### FR-RCV-001 · 送达场景

信件满足送达条件后，展示寄信人的记忆信使携带信封抵达的短动画。

该动画表达的是“一个人的视角抵达另一个人”，信封仅为传递载体，不得把送信包装为产品最终目标或游戏奖励。

### FR-RCV-002 · 主动开启

用户点击信封后才展开图文信件；不自动打开、不自动朗读。

### FR-RCV-003 · 动画控制

送达动画可跳过、可重看、无自动音效，并提供减少动态效果的静态版本。

### FR-RCV-004 · 阅读导航

支持按段落或页面阅读、查看插图、返回上次位置和进入结尾回应区。

### FR-TTS-001 · 听这封信

提供“听这封信”入口，服务不识字、低视力、年长或阅读疲劳用户。

### FR-TTS-002 · 朗读准确性

只朗读寄信人最终确认的文字，不在朗读时再次改写、概括或补充。

### FR-TTS-003 · 播放控制

支持播放、暂停、继续、重新播放、进度定位、语速调整和断点续听；当前文字可同步高亮。

### FR-TTS-004 · 声音选择

MVP提供少量温和中性、成熟女声、成熟男声；建议由收信人选择声音和语速。

### FR-TTS-005 · 画面描述

可选“同时描述画面”；画面说明作为辅助音轨，不混入寄信人正文。

# 16. 专属IP表情回应

### FR-EMO-001 · 回应定位

表情表达“怎样收下对方的故事”，不是开心、难过、点赞等普通聊天情绪。

### FR-EMO-002 · 视觉体系

使用《交换人生》IP或回应者自己的记忆信使完成动作；后续专项设计完整表情资产。

### FR-EMO-003 · 建议语义

至少覆盖轻轻收下、我看见了、想抱抱你、我会珍藏、陪你坐一会儿、我需要一点时间。

### FR-EMO-004 · 无文字发送

对方实际收到的是IP动作或微动画；选择界面可用名称或悬停提示帮助理解，但不要求发送文字。

### FR-EMO-005 · 发送规则

每次读信默认选择一个主要回应，可预览后发送；不支持连续刷表情、热度统计或评论区。

### FR-EMO-006 · 已读含义

发送专属表情同时确认已读；对暂时无法表达情绪的用户，提供中性的“轻轻收下”或“需要一点时间”。

### FR-EMO-007 · 结果保留

回应保留在信件结尾和对应交换记录中，双方可回看；默认不作为AI推断双方心理的依据。

# 17. AI视角交汇

## 17.1 生成条件与数据源

### FR-XVW-001 · 生成条件

必须存在双方最终确认并公开的信件，且双方均完成对方信件阅读确认，方可生成交汇。

### FR-XVW-002 · 单方无交汇

“我先告诉对方”场景中，B只发表情而未回信时不生成AI交汇；信件保存在已寄出/收到的信中。

### FR-XVW-003 · 唯一数据源

AI交汇只读取最终送达信件及用户主动公开附件，不读取私人草稿、原始录音、未公开图片、上下文摘要或被否定版本。

### FR-XVW-004 · 事实边界

不得裁判谁对谁错，不把未提及内容补成事实，不从沉默推断动机；所有推断性表达须清楚标明或避免。

## 17.2 共同事件交汇

### FR-XVW-005 · 时间节点

有明显顺序时按事件前、发生时、之后或关键阶段形成动态时间节点，不强制统一日期刻度。

### FR-XVW-006 · 事实线索

时间不明确时按共同地点、物件、行为、记忆细节和双方各自注意到的内容组织。

### FR-XVW-007 · 呈现结构

可包含“共同记得、各自看见、当时未被彼此看见”；后一项必须有信件明确依据。

## 17.3 一段人生交汇

### FR-XVW-008 · 主题交汇

按共同主题、人生阶段、相似感受、不同选择和来信唤起的记忆组织，不强行对齐成同一事实时间线。

### FR-XVW-009 · 结果名称

可使用“两段人生的回声”等叙事名称，与共同事件的“同一件事的两个视角”区分。

## 17.4 原句与确认

### FR-XVW-010 · 信件原句

允许引用双方最终信件中的关键原句，并标明来自哪一封信；不得引用原始私人表达。

### FR-XVW-011 · 结果查看

双方共同查看交汇结果，可跳转两封完整信件与专属表情；交汇不覆盖原信。

### FR-XVW-012 · 问题反馈

双方均可标记AI交汇中的事实错误或不准确关联；系统支持重新生成或隐藏有问题段落。

# 18. 内容权限与隐私边界

| **内容层级**       | **本人**           | **对方**       | **AI交汇**         |
|--------------------|--------------------|----------------|--------------------|
| 原始文字草稿       | 查看/编辑/删除     | 不可见         | 不可读取或引用     |
| 原始录音与原始转写 | 播放/校正/删除     | 默认不可见     | 不可读取或引用     |
| 原始图片           | 查看/删除/选择公开 | 主动公开后可见 | 仅公开后可使用     |
| AI上下文摘要       | 查看/纠错          | 不可见         | 不可读取或引用     |
| AI整理稿（未确认） | 查看/编辑          | 不可见         | 不可读取或引用     |
| 最终确认信件       | 查看               | 送达后查看     | 可读取与引用       |
| 最终确认插图       | 查看               | 送达后查看     | 可展示             |
| 专属表情回应       | 双方查看           | 双方查看       | 默认不用于心理推断 |
| AI视角交汇         | 双方查看           | 双方查看       | 共同结果           |

### FR-PRI-001 · 最小公开

默认只交换本人确认的故事、绘本及明确公开的附件；原始录音和图片必须由用户主动选择才可随结果公开。

### FR-PRI-002 · 来源可追溯

本人可回看文字、语音、图片如何进入整理稿；对方只看到最终公开版本，不看到整理过程。

### FR-PRI-003 · 删除提示

删除已经用于整理的原始材料前提示影响范围；已正式送达的共同内容需按双方共同数据规则处理。

# 19. 交汇记忆碎片

### FR-MEM-001 · 生成条件

一次交换完成双方信件、阅读确认和AI交汇后，系统生成双方共同拥有的交汇记忆碎片。

### FR-MEM-002 · 内容组成

包含事件或主题名称、事件时间、交换时间、两封最终信件、公开插图、专属表情、AI交汇和关键原句。

### FR-MEM-003 · 单方来信

只有一封来信与表情回应时不生成交汇记忆碎片，分别保存在A的已寄出和B的收到信件中；日后回信后可转为交汇记忆。

### FR-MEM-004 · 共同所有

记忆碎片在双方空间中保持同一身份和内容；任何影响共同内容的删除或隐藏操作须给出双方影响提示。

### FR-MEM-005 · 时间区分

同时记录故事发生时间与交换完成时间，避免把过去事件与重新讲述的时间混淆。

# 20. 共同记忆树与轻互动

## 20.1 记忆树入口与形态

### FR-TRE-001 · 共同入口

双人空间提供“记忆树”入口，用双方名字命名，如“丹与林夏的记忆树”，不以关系角色命名。

### FR-TRE-002 · 象征结构

视觉可采用共享土壤、根系交汇、树干各自生长的双生结构，表达理解但不覆盖彼此。

### FR-TRE-003 · 记忆节点

每块交汇记忆以叶片、果实、光点或藏在树上的信呈现，点击展开记忆卡并进入完整交汇。

### FR-TRE-004 · 状态生长

可将创建、表达中、单方寄出、双方送达、交汇完成分别表现为种子、芽、花苞、花和记忆果实，但不得暴露具体写作进度。

### FR-TRE-005 · 空状态

首次交换未完成时展示幼小的记忆树和轻提示，不制造完成压力。

## 20.2 共同浇水

### FR-WAT-001 · 非强制互动

双方可分别为某块记忆浇水，表达愿意照料这段被重新看见的记忆；不设任务、奖励或惩罚。

### FR-WAT-002 · 单方浇水

自己的信使完成浇水动作，自己一侧根系微亮；对方下次进入可看到有人浇过水，不发送催促通知。

### FR-WAT-003 · 双方浇水

双方无需同时在线；都完成后，两侧根系光在地下交汇，记忆节点亮起并长出新叶或花。

### FR-WAT-004 · 同时在线

双方恰好同时在线时可展示共同动画，但同时在线不得成为完成条件，也不显示精确在线监控。

### FR-WAT-005 · 不枯萎

长期不浇水时树不会枯萎，不设置连续天数、养分值、等级、排行榜和漏浇提醒。

## 20.3 可选轻互动

### FR-TRE-006 · 留下一束光

用户重访旧记忆时可留下微光，表示最近又想起这段记忆；对方看到状态但不产生回复义务。

### FR-TRE-007 · 原句回响

轻触节点可浮现最终公开信件中的一句原句，不调用私人草稿。

### FR-TRE-008 · 季节氛围

树可随现实季节产生视觉变化，但不影响内容和功能，不把季节变化与用户活跃度绑定。

# 21. 异常状态与恢复

| **场景**               | **处理要求**                                                         |
|------------------------|----------------------------------------------------------------------|
| 网络中断或页面关闭     | 本地与服务端分层保存；恢复后回到上次编辑位置，并提示最后成功保存时间 |
| 录音中断               | 保留已录片段，不自动丢弃；允许继续录音或提交现有片段转写             |
| 语音识别失败           | 保留原音，提示重试或手动补充；不得生成猜测文本                       |
| 图片识别失败           | 保留原图，允许用户添加说明或重新上传                                 |
| AI整理失败             | 原始材料不受影响；允许重试，不自动进入确认或寄出                     |
| AI插图失败             | 文字信件仍可继续；允许删除该画面、重试或使用原图                     |
| 预约时间到但一方未寄出 | 不单独送达，允许继续等待或重新约定                                   |
| 交汇生成错误           | 双方可标记问题，保留原信并允许重新生成交汇                           |
| 对方长期不回信         | 不催促、不显示未完成任务；来信保持可回看，回信入口长期保留           |
| 用户退出账号           | 私人内容不得在未登录状态暴露；重新登录后恢复对应空间                 |

# 22. Web端通用适配与验收标准

## 22.1 Web端通用要求

- 以桌面Web为核心呈现，不按手机页面直接拉伸；表达、整理、预览和阅读页充分利用横向空间。

- 核心操作在常见桌面视口中明确可见；固定输入区、AI面板和正文不得造成双滚动条冲突。

- 鼠标悬停可补充表情名称、按钮解释和节点信息，但关键功能不能只依赖悬停。

- 动画均支持跳过或减少动态效果；音频均由用户主动播放。

- 长文字、转写和图文信件须支持连续阅读、位置恢复和键盘操作。

- 所有表单和媒体操作提供清晰的加载、成功、失败、不可用和重试状态。

## 22.2 核心验收清单

- [ ] 用户可从品牌入口进入双入口选择页，并完成创建交换或邀请码加入。
- [ ] 四种“叙事内容×交换方式”组合均可被正确创建和执行。
- [ ] 共同事件卡由 A 创建、B 只读；一段人生使用有结束条件且可多次记录的私人时间线。
- [ ] 文字、语音、图片可以混合表达，原始材料不被AI覆盖。
- [ ] AI陪伴在用户表达时保持安静，只在主动唤醒或满足停顿条件时轻提示。
- [ ] AI 整理稿和个人绘本均需本人确认；共同绘本需双方分别确认，未经确认不得归档。
- [ ] 双方独立表达时，先寄出的信在另一方寄出前不可被阅读。
- [ ] 单方分享时，收信人可仅发表情，不被强制回信。
- [ ] 信件送达由记忆信使承接，用户主动点击后阅读，可主动开启语音读信。
- [ ] 专属表情既作为轻量回应，也作为已读确认，不形成聊天式互动。
- [ ] AI交汇只读取双方最终公开信件，不泄露任何私人原始表达。
- [ ] 只有完成双向信件与交汇后才生成共同记忆碎片并挂入记忆树。
- [ ] 双方可非强制地为记忆浇水；不活跃不会导致树枯萎或产生惩罚。

> **功能闭环**
>
> 创建关系 → 选择讲述范围与参与方式 → 建立事件卡/人生周期卡 → 单篇表达或持续记录 → AI 整理并确认源稿 → 单方生成个人绘本或双方提交后生成共同绘本 → 送达与阅读 → 轻回应 → 共同记忆。
