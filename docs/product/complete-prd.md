## 1. 产品概述

《交换人生》帮助两个彼此认识的人，把难以直接说清的经历整理成可以被对方完整阅读的故事。

用户可能想重新讲述一件双方共同经历的事，也可能想记录自己在远方的一段生活。表达过程中，用户可以主动开启 AI，让 AI 帮忙补足时间、人物、行为、感受和遗漏的上下文。AI 只负责提问、整理和生成候选内容，不判断谁对谁错，也不能替用户发布。

产品最终交付两类结果：

- 单方讲述：根据讲述者确认的内容，生成一册个人记忆故事和绘本，送给对方。
- 双方共述：双方分别完成自己的版本，在彼此不可见的情况下提交。两边都提交后，AI 基于双方确认公开的内容，生成一册保留差异的共同故事和绘本。

产品追求的是“让对方看见我经历了什么、当时怎样理解”，而不是强制和解。一次交换可以带来理解，也可能只是让差异第一次被完整地放在同一个空间里。

## 2. 背景与问题

亲密关系中的许多分歧不是因为缺少事实，而是双方只见过自己的那一面。普通聊天容易被即时反应打断；日记只属于自己；社交平台不适合承载私人关系；通用 AI 又缺少明确的发布边界和双人权限。

用户需要一个有节奏的表达空间：先独立讲完，再决定是否交给对方；需要 AI 帮忙梳理，但仍保留自己的措辞、犹豫和不确定；还需要一种比聊天记录更完整的结果，能够被阅读、回看和保存。

## 3. 产品目标

### 3.1 用户目标

1. 让用户能低压力地开始讲述，不要求一开始就写成完整文章。
2. 让用户掌握 AI 介入程度，并在发布前确认每一项 AI 结果。
3. 让单方分享可以独立成立，不把对方回写变成任务。
4. 让双人共述在互不干扰的前提下完成，最终同时看见两个视角及共同绘本。
5. 让一段较长的人生记录可以按天或按事件积累，最后形成双轨时间线。

### 3.2 产品目标

- 跑通“创建、邀请、表达、AI 整理、确认、生成、送达、阅读、保存”的闭环。
- 建立可解释的隐私边界：什么只有本人看见，什么已经提交，什么会被对方看见。
- 让共同事件和一段人生共用关系、权限和生成能力，但拥有各自适合的创作交互。
- 为比赛演示提供一条十分钟内可以讲清楚、同时具有情感辨识度和技术可信度的主流程。

### 3.3 非目标

- 不做陌生人匹配、公开社区或关系推荐。
- 不做即时聊天、评论区和无限表情流。
- 不判断谁正确，不提供心理诊断或关系评分。
- 不用签到、连续天数、排行榜或内容枯萎制造压力。
- 不允许 AI 自动寄出、自动替用户同意公开或代替另一方表达。

## 4. 用户与使用场景

### 4.1 主要用户

| 用户 | 典型关系 | 需要完成的事 | 主要顾虑 |
|---|---|---|---|
| 想解释的人 | 情侣、母女、父子、朋友 | 把一件有分歧的经历讲完整 | 怕表达再次变成争吵，怕 AI 改掉原意 |
| 想分享远方生活的人 | 异地亲人、异地伴侣 | 记录一段时间内的日常并在阶段结束后交给对方 | 内容零散、很难持续整理 |
| 收到邀请的人 | 上述关系中的另一方 | 决定是否参与共述，或只阅读对方的讲述 | 担心被迫回应，担心提前看到会影响自己的版本 |
| 年长或低数字熟练度用户 | 父母、祖辈 | 通过语音和图片完成表达、听取故事 | 不熟悉复杂编辑器，输入和阅读负担高 |

### 4.2 核心场景

#### 场景 A：共同事件，双方共述

两个人对一次争执、一次搬家、一次家庭决定有不同理解。发起人描述事件边界并邀请对方。双方分别写自己的版本，AI 在各自的私人空间里协助梳理。双方都确认提交后，AI 生成共同故事、双视角节点和绘本。双方看到彼此原文与共同结果，但共同结果不裁判对错。

#### 场景 B：共同事件，单方讲述

用户只想把自己当时的感受和理解说清楚。AI 帮助完成个人故事和绘本，内容确认后直接送给对方。对方可以阅读、留下一个“我收到了”的回应，也可以以后再写自己的版本。

#### 场景 C：一段人生，双方共述

异地生活的两个人建立一个有开始和结束日期的记录周期。每个人都能按天或按事件添加文字、语音、照片和地点说明，但看不到对方内容。周期结束后双方确认提交，AI 生成两条人生时间线及一本共同绘本。

#### 场景 D：一段人生，单方讲述

用户记录自己在远方的一段生活，周期内持续积累。结束时 AI 将日记、照片和语音整理为个人时间线和绘本，再送给家人。收信人不需要补写。

## 5. 产品模型

### 5.1 两个独立选择维度

创建时只做两个核心选择。界面需要直接说明选择结果，避免使用抽象术语。

| 维度 | 选项一 | 选项二 | 决定什么 |
|---|---|---|---|
| 讲述范围 | 同一件事 | 一段人生 | 创作空间、时间结构和 AI 整理方式 |
| 参与方式 | 只由我讲 | 邀请双方都讲 | 是否等待对方、生成个人绘本还是共同绘本 |

四种组合必须全部成立，不能用讲述范围推断参与方式。

### 5.2 组合与结果

| 编号 | 组合 | 创作方式 | 对方何时可见 | 最终结果 |
|---|---|---|---|---|
| M1 | 同一件事 + 只由我讲 | 一次性故事工作区 | A 确认寄出并满足送达条件后 | A 的个人故事、绘本和原始公开附件 |
| M2 | 同一件事 + 双方都讲 | 双方各自一次性工作区 | 双方提交后同时揭晓 | 两份确认故事、共同故事、双视角绘本 |
| M3 | 一段人生 + 只由我讲 | 有起止时间的连续记录 | 默认在 A 完成整段记录并寄出后 | A 的人生时间线、个人绘本 |
| M4 | 一段人生 + 双方都讲 | 双方各自连续记录 | 周期结束且双方提交后同时揭晓 | 双轨时间线、共同故事、共同绘本 |

### 5.3 核心对象

| 对象 | 定义 | 所有权 |
|---|---|---|
| 双人空间 | 两名用户长期共享的关系容器，可包含多次交换 | 双方可见，成员固定 |
| 交换项目 | 一次共同事件或一段人生计划 | 双方可见基础信息 |
| 事件卡 | 界定“同一件事”的中性背景 | A 创建确认，B 只读并确认识别 |
| 人生周期卡 | 定义主题、开始时间、结束时间和记录节奏 | A 创建；双人模式需 B 确认参与 |
| 私人素材 | 文字、录音、转写、照片、OCR 和 AI 工作摘要 | 始终属于上传者 |
| 故事源稿 | 用户确认、允许进入生成流程的叙事文本 | 提交前仅本人，提交后按模式公开 |
| 个人绘本 | 只基于一名用户确认内容生成的结果 | 作者拥有，送达后对方可读 |
| 共同绘本 | 只基于双方已提交内容生成的共同结果 | 双方共同查看，双方确认后归档 |
| 记忆节点 | 已完成交换在双人空间中的长期入口 | 单方结果分别保存；共同结果双方共有 |

## 6. 关键产品原则

1. 私密优先。草稿、原始录音、原图、转写、OCR 和 AI 工作内容默认只对本人可见。
2. AI 可选。用户不开启 AI 也能手工完成故事并寄出。
3. 先确认，再生成。AI 整理稿由用户确认后，才能进入故事和绘本生成。
4. 双方模式不抢跑。任何一方都不能在自己提交前看到对方正文；共同绘本必须等双方提交。
5. 单方模式能独立结束。收信人不回写，不影响个人故事送达和保存。
6. 共同结果保留分歧。AI 可以指出相同点和不同点，不能把两个版本改写成一个虚假的统一事实。
7. 表达与发布分离。上传、自动保存、AI 处理和“完成今天的记录”都不等于公开。

## 7. 端到端用户流程

### 7.1 创建流程

`进入产品 → 登录/注册 → 选择讲述范围 → 选择参与方式 → 填写事件卡或人生周期卡 → 设置送达规则 → 预览规则 → 创建项目 → 邀请对方或直接开始`

创建确认页必须用自然语言复述结果。例如：

> 你们会分别写下“那次搬家”的版本。在两个人都提交前，彼此看不到对方写了什么。两边提交后，系统会同时揭晓，并生成共同绘本。

用户需要在创建前看到以下信息：

- 对方是否必须参与；
- 内容在什么时候对对方可见；
- 是否可以在送达前取回；
- AI 会读取哪些内容；
- 最终生成的是个人绘本还是共同绘本。

### 7.2 邀请与加入

1. A 生成邀请链接和邀请码，通过外部渠道发送。
2. B 打开后先看到邀请人、项目类型、参与方式、时间范围和隐私说明，不显示 A 的故事内容。
3. 单方讲述模式下，B 可以直接加入双人空间等待来信。
4. 双方都讲模式下，B 明确点击“接受并写下我的版本”后加入。
5. B 若暂不接受，项目保持“等待回应”，A 只能看到未加入，不能看到拒绝原因。
6. 邀请过期、被使用、项目取消、自己打开自己的链接时给出明确恢复路径。

### 7.3 共同事件创作流程

`确认事件卡 → 私人表达 → 可选 AI 引导 → 完成表达 → AI/人工整理 → 确认故事源稿 → 选择公开图片 → 提交`

共同事件工作区围绕一篇完整叙事展开。推荐的 AI 引导顺序是：

1. 发生了什么；
2. 当时有哪些人在场；
3. 你当时注意到了什么；
4. 你当时怎么理解，后来有没有变化；
5. 哪些内容是事实，哪些是你的感受或猜测；
6. 最想让对方看见的是什么。

问题一次只出现一个，用户可以跳过。AI 不按固定问卷强迫用户补齐所有字段。

### 7.4 一段人生创作流程

`确认人生周期 → 进入私人时间线 → 多次新增记录 → 阶段回顾 → 周期结束 → 汇总整理 → 确认故事源稿 → 提交`

人生记录工作区不是一张长文本表单，而是一条私人时间线。每条记录包括：

- 日期或大致时间；
- 文字、语音、照片中的任意一种或多种；
- 可选地点；
- 当天标题或一句话；
- 私人标签；
- 是否纳入最终故事。

用户可随时新增、编辑、删除和排序自己的记录。双人模式中，对方只能看到“项目仍在记录中”或“对方已提交”，不能看到记录数量、日期和活跃情况。

周期结束后，AI 先生成个人时间线摘要和候选故事源稿。用户可以排除某些记录、修改顺序、补充结束语，再确认提交。

### 7.5 单方讲述生成与送达

`作者提交故事源稿 → 生成个人故事结构 → 作者确认 → 生成插图与版式 → 作者确认完整绘本 → 寄出 → 收信人开启 → 阅读/听取 → 轻回应或以后回写`

生成个人绘本时，AI 可以使用作者与 AI 的本次任务上下文，但只能使用作者明确纳入最终故事的内容。被删除、排除或仅作私人参考的素材不得进入生成。

### 7.6 双方共述生成与揭晓

`A 提交并托管 → B 提交并托管 → 锁定双方公开快照 → AI 直接生成共同故事与绘本 → 双方校对并确认 → 阅读共同绘本 → 按访问策略决定是否开放双方源稿 → 归档共同记忆`

双方提交后默认先进入共同绘本，不先阅读彼此源稿。项目保存 `source_access_policy`：`hidden` 表示双方始终不能在产品中阅读彼此源稿；`after_storybook` 表示用户完成共同绘本阅读后，才开放源稿入口。比赛版默认使用 `after_storybook`，同时保留切换到 `hidden` 的配置能力。

共同故事至少保留四类内容：

- 双方共同确认或共同提到的线索；
- A 的视角；
- B 的视角；
- 仍然不同或无法确认的地方。

共同故事中的事实性句子必须能追溯到 A、B 或双方的公开源稿。AI 不得为了让故事更顺滑而虚构对白、动机或和解结局。

任一方认为共同结果有误时，可以标记具体段落并选择“事实不准确、混淆了双方、语气不属于我、这段不适合进入共同绘本”。问题段落在处理前不进入最终归档。

### 7.7 阅读与回应

收信人主动打开后阅读图文故事，可切换纯文字、图文分页和语音朗读。阅读进度自动保存。

读完后的操作按轻重排序：

1. 轻轻收下；
2. 我看见了；
3. 想抱抱你；
4. 我需要一点时间；
5. 我也想写下我的版本。

前四项是一次性回应，不形成聊天。最后一项创建与原项目关联的新表达或把单方项目升级为双人共述，但必须重新说明可见性和生成规则。

## 8. 信息架构与页面

### 8.1 顶层导航

登录后只保留四个稳定入口：

- 首页：当前项目、待处理动作和最近记忆；
- 双人空间：与某位关系对象共同创建的所有项目；
- 我的故事：本人草稿、已寄出和私人素材；
- 设置：账号、AI 偏好、通知、隐私和数据管理。

### 8.2 核心页面清单

| 页面 | 目标 | 核心操作 |
|---|---|---|
| 品牌入口 | 建立氛围并进入产品 | 进入 |
| 登录/注册 | 建立可恢复身份 | 登录、注册、回到原目标 |
| 首页 | 告诉用户现在最该做什么 | 继续记录、查看邀请、开启来信 |
| 新建项目 | 完成两个核心选择 | 选范围、选参与方式 |
| 事件卡 | 界定共同事件 | 描述、AI 中性整理、确认 |
| 人生周期卡 | 设定一段人生 | 主题、起止时间、记录方式、确认 |
| 创建确认 | 说明完整规则 | 返回修改、创建 |
| 邀请预览 | 让 B 知道自己将参与什么 | 接受、稍后、退出 |
| 项目空间 | 呈现双方粗粒度状态 | 继续表达、查看规则、管理邀请 |
| 故事工作区 | 完成共同事件表达 | 写、录、上传、开启 AI |
| 人生时间线 | 连续积累记录 | 新增记录、编辑、阶段回顾 |
| AI 引导侧栏 | 提供可控的提问和整理 | 开启、暂停、跳过、查看上下文 |
| 整理确认 | 校对 AI 候选源稿 | 差异对照、编辑、确认 |
| 内容选择 | 决定哪些材料公开 | 选择记录、原图和附件 |
| 绘本编辑 | 调整结构、插图和版式 | 换图、删图、编辑标题、确认 |
| 提交/寄出确认 | 明确公开后果 | 确认提交、取回规则 |
| 等待与揭晓 | 解释当前状态 | 返回首页、到期处理、同时揭晓 |
| 故事阅读 | 完整接收故事 | 阅读、听取、查看来源 |
| 共同校对 | 双方处理共同 AI 结果 | 标记问题、确认、重生成 |
| 记忆详情 | 长期回看最终结果 | 阅读原故事、共同绘本、回应 |

### 8.3 当前交互需要调整的地方

| 当前问题 | 用户风险 | 调整方案 |
|---|---|---|
| “人生片段”仍按一次性长文处理 | 无法支撑数天或数周的异地记录 | 增加人生周期卡和私人时间线工作区 |
| 创建页先讲抽象概念 | 用户不知道选择后会发生什么 | 用场景语言和结果预览解释每个选项 |
| AI 引导入口和 AI 生成边界混在一起 | 用户不清楚 AI 读了什么 | 首次开启时显示本次读取范围，可随时暂停 |
| 现有结果以信件和“视角交汇”为中心 | 与“故事绘本”主价值不一致 | 把故事源稿、个人/共同绘本设为主输出，信封只负责送达 |
| 双方模式缺少共同绘本的双人校对 | AI 结果可能代表错人或提前归档 | 增加共同校对、问题标记和双方确认状态 |
| 人生记录没有结束和未完成处理 | 一方长期不提交时项目无法收束 | 提供延期、按现有内容结束、转为单方保存、取消四种处理 |
| 邀请加入过于直接 | B 可能在不了解任务时被绑定 | 双方模式增加邀请预览和明确接受动作 |
| 阅读、表情、是否回写分成过多跳转 | 情绪刚建立就被页面切断 | 在阅读结尾用一层底部面板完成回应和回写选择 |
| 发送规则到最后才出现 | 用户写完才发现要等待或同时公开 | 创建确认时先讲清，寄出时再次确认 |
| 首页按功能入口组织 | 用户难以判断下一步 | 按“需要你处理、正在进行、已经完成”组织项目 |

## 9. 详细功能需求

### 9.1 账号与关系空间

- 支持邮箱魔法链接或验证码登录，登录后回到原业务目标。
- 首次使用只要求昵称和头像；关系标签可选且不影响权限。
- 同一对用户可以拥有一个双人空间，空间中可以创建多次交换项目。
- 用户可以同时与不同熟人建立多个独立空间，内容不得跨空间混用。
- 关系空间解除、拉黑和重新邀请规则需要独立设计，MVP 至少支持退出未开始项目。

### 9.2 创建项目

- 创建过程保存为草稿，刷新后恢复。
- 两个核心选择必须分开存储，不允许根据项目类型自动锁定参与方式。
- 共同事件至少包含标题和中性描述；时间、地点和人物可选。
- 一段人生至少包含主题、开始时间和结束条件。结束条件可以是日期或“由我手动结束”。
- 双人模式必须显示“提交前互不可见”和“双方提交后同时揭晓”。
- 单方模式必须显示“对方不需要回写”。
- 创建后允许 A 在 B 加入前修改设置；B 接受后，影响可见性和截止时间的修改需双方确认。

### 9.3 私人表达

- 支持文字、语音和图片混合输入。
- 所有输入自动保存，并显示最后成功保存时间。
- 网络断开时先保存在本地，恢复后处理版本冲突。
- 原始文字、编辑稿、AI 整理稿分别保存，AI 不覆盖原稿。
- 用户可以手动将某段标记为“仅给自己”“可进入故事”“可作为公开原图”。
- “完成本次记录”只结束一条人生记录；“提交整个项目”才会进入公开和生成流程。

### 9.4 AI 引导

- AI 默认关闭，用户主动开启后才建立本次引导会话。
- 开启前显示本次将读取的内容类型和用途。
- 用户可选择“帮我补全故事”“帮我理清顺序”“陪我说下去”“帮我区分事实和感受”。
- AI 一次只问一个问题，允许跳过、返回和暂停。
- AI 可以复述用户已经提供的内容，但要标明这是整理，不是对事实的判断。
- AI 不推断对方动机，不建议用户应该原谅、分手或作出其他关系决定。
- 遇到自伤、暴力或现实人身安全风险时，停止普通叙事引导，展示求助建议；具体策略需在上线前由安全方案确认。

### 9.5 语音与图片

- 录音必须由用户主动开始，不因停顿自动结束。
- 转写保留原话版和可编辑书面版，低置信内容可定位回听。
- 原音默认不公开；用户可以只公开整理后的文字。
- 图片上传后允许补充日期、地点和“它为什么重要”。
- OCR 和图片描述只作为私人候选信息，不能自动进入故事。
- AI 插图必须标记为生成内容，不把生成画面描述成真实照片。
- 未获得授权时，不生成可识别的真实人物面孔。

### 9.6 故事整理与确认

- AI 只能整理用户选择纳入的材料。
- 整理允许调整顺序、分段、标点和重复口语，不允许补写事实。
- “大概、可能、我记不清”等不确定表达必须保留。
- 用户可以并排查看原始表达和整理稿，并定位 AI 改动。
- 用户修改已确认源稿后，相关摘要、插图和绘本预览全部标记为过期。
- 用户可以跳过 AI 整理，直接把人工编辑稿确认为故事源稿。

### 9.7 个人故事与绘本

- 个人故事结构由标题、引子、时间线或章节、正文、结束语组成。
- 用户可选择文字为主、图文平衡或图片为主，但任何模式都不能静默删除已确认文字。
- AI 先提出候选画面，再由用户选择是否生成。
- 用户可删除、重新生成、替换为公开原图或调整图片位置。
- 绘本预览必须与收信人实际看到的版本一致。
- 用户最终确认后才能寄出；确认记录包含内容版本和附件清单。

### 9.8 共同故事与绘本

- 生成前服务端重新校验双方都已提交，并锁定两份公开源稿快照。
- 输出按共同事件和一段人生使用不同结构。
- 共同事件突出同一节点中的不同注意、理解和感受。
- 一段人生展示两条独立时间线，可按日期、阶段或主题产生交叉节点。
- 每一段共同叙述在系统内部保留来源引用。只有 `source_access_policy=after_storybook` 且当前用户已完成共同绘本阅读时，界面才允许跳回 A 或 B 的源稿；`hidden` 模式只用于事实校验和问题反馈，不向用户开放原文入口。
- 双方都能标记问题。重生成不能修改两份源稿。
- 共同绘本只有在双方确认后进入共同记忆；一方暂不确认时保留为待校对状态。
- 为避免无限僵持，MVP 可允许双方分别保存自己的源稿和当前共同候选，但不能把未共同确认的版本标为“共同记忆”。

### 9.9 提交、等待与送达

- 单方模式下，作者确认个人绘本后立即或预约送达。
- 双方模式下，先提交的一方进入托管状态，不能再看到或修改对方状态细节。
- 送达前可取回自己的提交；取回后旧快照失效，项目回到表达中。
- 双方项目有结束日期时，到期未完成不自动公开任何一方内容。
- 到期处理提供延期、继续等待、各自保存、由已提交者转为单方分享。最后一种会改变公开规则，必须由已提交者再次确认，且不能公开另一方内容。
- 任何送达、揭晓和生成操作都必须幂等，重复点击不能创建多个结果。

### 9.10 阅读、回应和后续表达

- 阅读页支持图文分页、纯文字、语音朗读、字号和语速调整。
- 收信人可以查看哪些图片是原图、哪些是 AI 插图。
- 双方模式默认先阅读共同绘本；源稿入口由 `source_access_policy` 和当前用户的绘本阅读完成状态共同决定。
- 阅读位置保存在当前用户名下，不向寄信人暴露精确进度。
- 寄信人只看到未开启、已开启、已回应三类粗粒度状态。
- 一封故事只保留一个主要回应，允许在短时间内修改一次，不统计热度。
- “写下我的版本”创建关联表达，并重新经过私人表达、确认和送达流程。

### 9.11 通知与提醒

- 通知只用于邀请、到期前提醒、对方已提交、内容已送达、共同结果待校对和 AI 任务失败。
- 双人模式中，“对方已提交”可以显示，但不能显示字数、记录数量或提交时间线细节。
- 不发送“对方在等你”“你已经几天没写”等带压力的提醒。
- 用户可关闭邮件通知，产品内仍保留状态入口。

### 9.12 记忆与回看

- 已完成项目按发生时间和交换完成时间双重归档。
- 单方项目显示个人绘本、公开源稿和对方回应。
- 双方项目显示两份源稿、共同故事、共同绘本和双方问题处理记录。
- 私人素材不自动进入共同记忆，也不能被对方搜索。
- 记忆树属于后续增强能力；MVP 先使用时间轴或书架完成可靠检索和回看。

## 10. 状态设计

### 10.1 项目状态

| 状态 | 含义 | 用户可执行动作 |
|---|---|---|
| 配置中 | A 尚未创建完成 | 修改、退出 |
| 等待加入 | 邀请已创建，B 尚未加入 | 复制邀请、取消 |
| 记录中 | 至少一人开始表达 | 继续写、管理自己的素材 |
| 待本人确认 | 故事源稿或个人绘本等待本人确认 | 编辑、重生成、确认 |
| 单方待送达 | 单方内容已确认，等待立即或预约送达 | 取回、查看规则 |
| 双方等待 | 一方已提交，等待另一方 | 取回、继续等待 |
| 待揭晓 | 双方已提交，等待预约时间或生成任务 | 查看状态 |
| 共同生成中 | 正在生成共同故事或绘本 | 查看进度、失败后重试 |
| 待共同校对 | 共同结果等待双方确认 | 确认、标记问题 |
| 已送达 | 个人结果已送达，或双方内容已揭晓 | 阅读、回应 |
| 已完成 | 最终结果已归档 | 回看、导出允许的内容 |
| 已取消 | 项目终止且未公开 | 查看私人内容处理结果 |

### 10.2 人生记录状态

| 状态 | 含义 |
|---|---|
| 草稿 | 一条记录尚未完成 |
| 已保存 | 记录保存在私人时间线中 |
| 已排除 | 保留在私人空间，但不进入最终故事 |
| 待整理 | 周期结束，等待汇总 |
| 已纳入 | 用户确认进入故事源稿 |
| 已锁定 | 已随故事源稿提交，形成公开快照 |

### 10.3 AI 任务状态

所有 AI 能力使用 `idle → queued → running → review_required → confirmed`，失败分支为 `failed` 或 `timeout`。失败不得改变项目公开状态，也不得覆盖用户原始内容。

## 11. 权限与隐私

| 内容 | 本人 | 对方 | 私人 AI | 共同生成 AI |
|---|---|---|---|---|
| 草稿与人生记录 | 读写 | 不可见 | 本人开启后可读选中内容 | 不可读 |
| 原始录音和转写 | 读写 | 默认不可见 | 本人触发后可读 | 不可读 |
| 原始图片和 OCR | 读写 | 仅主动公开部分 | 本人触发后可读 | 仅公开附件快照 |
| AI 引导上下文 | 查看、清除 | 不可见 | 当前任务可用 | 不可读 |
| 已确认故事源稿 | 查看 | 满足送达或揭晓条件后查看 | 不默认复用 | 双方模式提交后可读固定快照 |
| 个人绘本 | 查看 | 送达后查看 | 作者任务可用 | 不用于另一方私人整理 |
| 共同故事和绘本 | 双方查看 | 双方查看 | 不可反向读取私人材料 | 共同任务结果 |

权限执行要求：

- 所有公开动作在服务端按内容版本记录确认人、确认时间、用途和附件范围。
- 浏览器不能持有 Supabase Service Role Key 或 AI 服务密钥。
- 文件存储使用私有桶和短时签名地址。
- 普通日志不记录完整故事、录音、OCR 和 AI 上下文。
- 用户删除私人材料时，同时清理对应 AI 工作副本；已公开结果按撤回规则处理。
- 共同生成前固定来源快照，禁止在 Prompt 中混入未公开版本。

## 12. AI 能力设计

### 12.1 Agent 划分

| Agent | 输入 | 输出 | 是否需要确认 |
|---|---|---|---|
| 引导 Agent | 本人选定的当前草稿和前置卡 | 单个问题、事实/感受梳理、上下文回顾 | 问题无需确认，内容不自动写入 |
| 整理 Agent | 本人选定的文字、转写和图片说明 | 故事源稿候选 | 本人确认 |
| 个人故事 Agent | 本人确认源稿和公开附件 | 个人故事结构、标题和画面建议 | 本人确认 |
| 共同故事 Agent | 双方提交后的公开源稿快照 | 共同线索、双视角段落、差异项 | 双方校对 |
| 插图 Agent | 已确认的画面描述和人物限制 | AI 插图候选 | 个人图由本人确认，共同图由双方确认 |
| 朗读服务 | 已送达最终文字 | TTS 音频和时间轴 | 用户主动播放 |

### 12.2 AI 引导规则

- 用户明确开启、暂停和结束 AI 会话。
- 每次提问说明是为了补充故事，不将问题伪装成心理结论。
- AI 避免“你其实”“对方可能是因为”等动机推断。
- AI 发现前后矛盾时使用“这里出现了两个不同时间，你想保留哪一种表达”，不替用户决定。
- AI 允许保留空白。故事不完整也可以由用户确认提交。
- 对于攻击、羞辱或指控性内容，AI 可以提示用户区分观察、感受和判断，但不能审查掉用户的真实经历。涉及违法和现实安全风险时按安全策略处理。

### 12.3 共同生成规则

- 输出不得使用“真相是”“其实你们都”等裁判式口吻。
- 同一事实有冲突时，分别写为“A 记得……，B 记得……”。
- 没有来源的细节不生成；模型不确定时返回缺口而不是补写。
- 每个段落保留来源 ID，系统支持逐段查看来源。
- 共同故事不默认写成和解结局，可以停在“双方仍有不同理解”。

## 13. 异常与恢复

| 场景 | 处理 |
|---|---|
| 自动保存失败 | 保留本地版本，显示最后成功保存时间，恢复后合并或让用户选择版本 |
| 录音中断 | 保留已录部分，不自动丢弃 |
| 转写或 OCR 失败 | 保留原素材，允许重试和人工补充 |
| AI 引导失败 | 用户继续手工写，不阻塞创作 |
| 故事整理失败 | 保留原稿和上一个候选版本，允许重试 |
| 插图失败 | 允许使用纯文字、原图或稍后生成 |
| 一方长期未加入 | A 可取消或将项目改为单方模式；修改前再次确认公开规则 |
| 周期结束一方未提交 | 不公开已提交内容；允许延期、继续等、各自保存或单方转化 |
| 共同结果一方不确认 | 保留两份已揭晓源稿，共同候选不进入共同记忆 |
| 邀请链接泄露 | 邀请仅可由一个非发起账号领取，A 可在领取前使其失效 |
| 账号注销 | 私人内容删除；共同内容保留与署名方式按待确认规则执行 |

## 14. 非功能要求

### 14.1 性能

- 常用页面在正常网络下首屏可交互时间目标小于 2.5 秒。
- 文本保存采用防抖和版本号，连续输入不阻塞。
- 大文件上传可暂停、重试并显示进度；上传失败不丢失其他草稿。
- AI 长任务异步执行，用户离开页面后仍可回来查看结果。

### 14.2 可用性与无障碍

- 桌面 Web 为主要体验，同时保证常见平板与手机可完成关键流程。
- 年长用户路径优先提供语音输入、明确按钮、可调字号和语音朗读。
- 所有动画可跳过并支持减少动态效果。
- 颜色不是唯一状态提示；键盘可完成主要表单和阅读操作。
- 关键按钮使用明确动词，如“提交我的版本”，避免只写“下一步”。

### 14.3 安全与合规

- 身份、成员关系、内容版本和送达状态必须由服务端校验。
- Supabase 使用行级安全策略隔离双人空间和私人草稿。
- 敏感文件使用短时访问地址，不提供可猜测的公开 URL。
- 记录 AI 任务来源、用途、版本和结果状态，但不在普通日志保存正文。
- 正式上线前需补充用户协议、隐私政策、AI 生成标识、未成年人和内容安全规则。

## 15. 数据与指标

比赛和早期测试阶段只采集产品验证所需的最小事件，不采集故事正文作为分析数据。

| 目标 | 指标 | 说明 |
|---|---|---|
| 用户能开始 | 创建完成率 | 从选择项目到成功创建 |
| 用户能表达 | 首条有效内容完成率 | 至少保存文字、语音或图片之一 |
| AI 有帮助 | AI 候选采纳率 | 确认稿中保留的 AI 整理比例，只记录数值不记录正文 |
| 双人流程成立 | 邀请接受率、双方提交率 | 分项目类型统计 |
| 结果被接收 | 送达开启率、阅读完成率 | 不记录逐字阅读行为 |
| 共同生成可信 | 问题标记率、重生成率、双方确认率 | 用于发现共同 Agent 质量问题 |
| 长周期可持续 | 有效记录天数、中途退出率 | 不做连续天数奖励 |

## 16. MVP 与阶段规划

### 16.1 比赛版 MVP

比赛版只做一条主流程和一条差异化展示：

1. 主流程：共同事件 + 双方都讲。
2. 差异化展示：一段人生 + 双方都讲的双轨时间线，可使用预置多日数据演示。
3. 必做输入：文字；语音转写和图片上传至少有一个能真实运行，另一个可受控模拟。
4. 必做 AI：私人引导、故事整理、共同故事生成、三至五幅绘本插图。
5. 必做规则：邀请码、互不可见、双方提交、来源快照、共同校对。
6. 必做结果：两份源稿、共同故事、共同绘本、长期回看入口。

单方模式应保留完整产品规则，并至少通过一套可演示流程证明它能独立结束。记忆树、信使测试、复杂预约、实体印刷和完整 RAG 不进入比赛版核心范围。

### 16.2 正式 MVP

- 真实 Supabase Auth、双人空间和邀请绑定；
- 共同事件四种组合；
- 一段人生的周期、记录、结束和异常收束；
- 文字、语音、图片的真实保存与处理；
- 个人和共同绘本生成；
- 阅读、回应、通知、隐私和数据删除基础能力；
- 服务端状态机、RLS、幂等和审计。

### 16.3 后续增强

- 实体绘本导出与印刷；
- 更丰富的绘本风格和人物一致性；
- 共同记忆书架或时间轮盘；
- 经明确授权的私人记忆检索；
- 多语言和方言增强；
- 关系空间长期管理与多个记录周期。

## 17. 核心验收标准

- [ ] 用户能理解两个选择各自决定什么，并在创建前知道内容何时公开。
- [ ] 四种组合都能创建，且单方模式不要求对方回写。
- [ ] 共同事件使用完整故事工作区，一段人生使用多条记录组成的私人时间线。
- [ ] AI 只能在用户开启后读取授权内容，关闭 AI 后仍能完成主流程。
- [ ] 草稿、原音、原图和 AI 工作上下文不会被对方读取。
- [ ] 双方模式中，任何一方提交前都看不到对方的故事或具体进度。
- [ ] 单方模式只生成作者个人故事和绘本，确认后可独立送达。
- [ ] 双方模式必须等两边提交后才生成共同故事和绘本。
- [ ] 共同结果保留来源、差异和不确定性，双方可逐段反馈。
- [ ] 未经双方确认的共同候选不得归档为共同记忆。
- [ ] 一段人生到期但有人未完成时，不会自动泄露已提交内容。
- [ ] AI、上传或保存失败不会覆盖原始内容，也不会错误推进公开状态。

## 18. 尚未梳理清楚、必须单独决策的问题

以下问题不会阻止比赛原型，但会直接影响正式产品的交互、数据结构或隐私规则。

已确认：双方提交后不先揭晓源稿，系统直接生成共同故事与绘本。源稿使用 `hidden/after_storybook` 访问策略控制。

### P0：进入开发前必须决定

| 问题 | 当前建议 | 不决定的影响 |
|---|---|---|
| 产品主承诺是否使用“和解” | 使用“理解、看见、重新讲述”，不承诺和解 | AI 语气、品牌文案和验收目标会摇摆 |
| 双方模式是否必须由 B 明确接受 | 必须接受后才绑定为共同创作者 | 受邀人可能在不知情时承担写作任务 |
| 一段人生的默认结束方式 | 必须有日期或手动结束条件，双人模式由双方确认延期 | 无法定义生成时点和未完成处理 |
| 共同绘本由谁确认 | 双方分别确认；未双确认不进入共同记忆 | AI 可能替未同意的一方“定稿” |
| 到期一方未提交怎么办 | 默认不公开；允许延期、各自保存或已提交方转单方 | 先提交者可能被无限等待或被意外公开 |
| AI 上下文能保留多久 | 按单次项目保存，用户可清除；普通日志不存正文 | 隐私说明和数据删除无法实现 |
| 已送达内容能否撤回 | 建议不能让对方“失忆”，可停止后续共享并标记作者撤回 | 共同数据的删除和双方权益无法定义 |

### P1：正式 MVP 前决定

| 问题 | 建议方向 | 需要补充的功能细节 |
|---|---|---|
| 单方人生记录是否支持边记录边分享 | 首版只在阶段结束后统一送达 | 若支持，需要逐条公开状态、订阅和撤回规则 |
| 人生周期最长多久 | 首版建议 1 天至 90 天 | 影响存储、提醒、AI 上下文和生成成本 |
| 每条记录和媒体上限 | 按产品成本设置软限制并显示剩余量 | 影响上传、处理队列和异常提示 |
| 是否支持真实人物一致性插图 | 首版不生成可识别真人，使用非写实角色 | 后续需要肖像授权、参考图保存和未成年人规则 |
| 共同结果发生争议怎么办 | 可标记段落、隐藏候选、重生成；不允许单方改写对方 | 需要争议状态、通知和版本历史 |
| 单方项目升级为双人项目 | 允许，但 B 从收到内容后写，不能再称为“盲写” | 需要区分“看后回写”和“互不可见共述” |
| 账号注销后共同记忆怎么处理 | 保留共同内容，作者匿名化或显示已注销 | 需要双方通知和导出政策 |
| 未成年人能否使用 | 正式上线前明确年龄门槛和监护规则 | 影响注册、数据处理和内容安全 |
| 商业模式 | 比赛版不收费；后续可按高清绘本/实体印刷收费 | 影响生成额度、重试次数和导出质量 |

### P2：体验增强阶段决定

| 问题 | 备选方向 |
|---|---|
| “时间轮盘”最终形态 | 双轨线性时间轴、可旋转时间轮、绘本章节书架 |
| 记忆信使是否保留 | 若不能提升表达和送达理解，降为轻量品牌角色 |
| 是否支持 PDF/图片/视频导出 | 优先 PDF，其次长图；视频成本和隐私风险更高 |
| 是否支持实体印刷 | 需处理地址、支付、高清生成、内容审核和物流 |
| 是否加入私人 RAG | 只在用户明确授权的本人记忆中检索，与共同索引隔离 |

## 19. 对现有 Demo 的改造结论

现有 Demo 可以继续作为共同事件基础流程，但正式产品不应直接在其上堆功能。迁移到 Next.js 时建议按以下顺序改造：

1. 保留现有共同事件写作、整理、预览和阅读组件。
2. 把 `narrativeType` 从“共同事件/人生片段”明确为“共同事件/人生周期”。
3. 新建人生时间线数据模型和工作区，不复用单篇长文本草稿。
4. 把“信件”从最终产物调整为送达包装，核心实体改为故事源稿、个人绘本和共同绘本。
5. 在双人模式中加入提交快照、双方揭晓、共同生成、共同校对和双确认。
6. 将 AI 引导改为显式开启，并展示本次数据使用范围。
7. 用服务端状态机和 Supabase RLS 替换浏览器本地状态中的权限判断。

完成以上调整后，产品叙事、交互流程和 Next.js + Supabase 架构才会指向同一套业务模型。
